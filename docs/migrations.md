# Migration Writing Guide

This document covers everything you need to write, read, and customise migration files for the Go migration framework.

Migrations are auto-generated by `makemigrations makemigrations`, but you will often need to:
- Understand what a generated file does
- Modify a generated migration (add a data backfill, change a field option)
- Write a migration entirely by hand (complex DDL, RunSQL, data seeding)
- Understand why a particular operation was generated

---

## Anatomy of a Migration File

Every migration is a regular Go source file in `package main`. It registers itself with the global migration registry using `func init()`, which Go calls automatically when the binary starts.

```go
package main

import (
    m "github.com/ocomsoft/makemigrations/migrate"
)

func init() {
    m.Register(&m.Migration{
        Name:         "0002_add_phone",
        Dependencies: []string{"0001_initial"},
        Operations: []m.Operation{
            &m.AddField{
                Table: "users",
                Field: m.Field{Name: "phone", Type: "varchar", Length: 20, Nullable: true},
            },
        },
    })
}
```

**Key points:**

- `package main` — all files in `migrations/` are in the same package
- `import m "..."` — the `migrate` package is aliased `m` by convention
- `func init()` — called automatically; no `main()` required in migration files
- `m.Register(...)` — adds the migration to the global registry
- `Name` — must be unique across all migrations; conventionally `NNNN_description`
- `Dependencies` — names of migrations that must be applied before this one
- `Operations` — ordered list of schema changes to apply

The `init()` pattern means migration files are self-registering. When the `migrate` binary starts, all `init()` functions run in alphabetical file order, populating the registry before any command executes.

## The `Migration` Struct

```go
type Migration struct {
    Name         string      // Unique identifier, e.g. "0001_initial"
    Dependencies []string    // Names of migrations this depends on
    Operations   []Operation // Schema operations to apply, in order
    Replaces     []string    // Squash only: names of migrations this replaces
}
```

### `Name`

The unique identifier for this migration. The `makemigrations makemigrations` command uses the format `NNNN_description` (e.g. `0003_add_user_preferences`). You can use any string, but keep names:
- **Unique** — duplicate names cause a panic at startup
- **Descriptive** — they appear in `status` output and the history table
- **Stable** — once applied in production, never rename a migration

### `Dependencies`

Names of migrations that must be fully applied before this one runs. The DAG is built from these links. An empty slice means the migration has no prerequisites.

```go
// No dependencies (first migration)
Dependencies: []string{},

// Single linear dependency
Dependencies: []string{"0001_initial"},

// Merge migration: depends on two branch leaves
Dependencies: []string{"0002_branch_a", "0003_branch_b"},
```

### `Operations`

An ordered slice of `Operation` values. They are applied top-to-bottom on `up` and reversed bottom-to-top on `down`. See [All Operations](#all-operations) below.

### `Replaces`

Only used in squashed migrations. Lists the names of the original migrations that this squash replaces. See [Squash Migrations](#squash-migrations).

## The `Field` Struct

`Field` describes a single database column. It is used in `CreateTable`, `AddField`, and `AlterField` operations.

```go
type Field struct {
    Name       string
    Type       string
    PrimaryKey bool
    Nullable   bool
    Default    string
    Length     int
    Precision  int
    Scale      int
    AutoCreate bool
    AutoUpdate bool
    ForeignKey *ForeignKey
    ManyToMany *ManyToMany
}
```

### Field Properties

| Property | Type | Description |
|----------|------|-------------|
| `Name` | `string` | Column name. Required. |
| `Type` | `string` | Column type. See [Field Types](#field-types) below. |
| `PrimaryKey` | `bool` | If `true`, column is the primary key. |
| `Nullable` | `bool` | If `true`, column allows NULL. Default `false` means NOT NULL. |
| `Default` | `string` | Default value reference name (e.g. `"now"`, `"true"`, `"new_uuid"`). |
| `Length` | `int` | Max character length, used with `varchar`. |
| `Precision` | `int` | Total significant digits, used with `decimal`/`numeric`. |
| `Scale` | `int` | Digits after the decimal point, used with `decimal`/`numeric`. |
| `AutoCreate` | `bool` | Auto-set to current timestamp on row creation (`created_at` pattern). |
| `AutoUpdate` | `bool` | Auto-set to current timestamp on row update (`updated_at` pattern). |
| `ForeignKey` | `*ForeignKey` | Adds a foreign key constraint. See [ForeignKey](#foreignkey). |
| `ManyToMany` | `*ManyToMany` | Declares a many-to-many relationship. See [ManyToMany](#manytomany). |

### Field Types

The `Type` string maps to the provider's SQL dialect. Common values:

| Type | PostgreSQL | MySQL | SQLite | Notes |
|------|-----------|-------|--------|-------|
| `"integer"` | `INTEGER` | `INT` | `INTEGER` | |
| `"biginteger"` | `BIGINT` | `BIGINT` | `INTEGER` | |
| `"smallinteger"` | `SMALLINT` | `SMALLINT` | `INTEGER` | |
| `"varchar"` | `VARCHAR(n)` | `VARCHAR(n)` | `TEXT` | Set `Length` |
| `"text"` | `TEXT` | `TEXT` | `TEXT` | |
| `"boolean"` | `BOOLEAN` | `TINYINT(1)` | `INTEGER` | |
| `"decimal"` | `DECIMAL(p,s)` | `DECIMAL(p,s)` | `REAL` | Set `Precision`/`Scale` |
| `"float"` | `FLOAT` | `FLOAT` | `REAL` | |
| `"timestamp"` | `TIMESTAMP` | `DATETIME` | `TEXT` | |
| `"date"` | `DATE` | `DATE` | `TEXT` | |
| `"uuid"` | `UUID` | `CHAR(36)` | `TEXT` | |
| `"json"` | `JSONB` | `JSON` | `TEXT` | |
| `"foreign_key"` | `INTEGER` + FK | `INTEGER` + FK | `INTEGER` + FK | Pair with `ForeignKey` |

### ForeignKey

```go
type ForeignKey struct {
    Table    string // Referenced table name
    OnDelete string // CASCADE, SET NULL, RESTRICT, etc.
    OnUpdate string // CASCADE, SET NULL, RESTRICT, etc.
}
```

Example:

```go
m.Field{
    Name: "user_id",
    Type: "foreign_key",
    ForeignKey: &m.ForeignKey{
        Table:    "users",
        OnDelete: "CASCADE",
    },
},
```

### ManyToMany

```go
type ManyToMany struct {
    Table string // The junction table name
}
```

Example:

```go
m.Field{
    Name:       "tags",
    Type:       "many_to_many",
    ManyToMany: &m.ManyToMany{Table: "post_tags"},
},
```

## The `Index` Struct

```go
type Index struct {
    Name   string   // Index name (must be unique in the database)
    Fields []string // Column names included in the index
    Unique bool     // If true, creates a UNIQUE index
}
```

Example:

```go
m.Index{Name: "idx_users_email", Fields: []string{"email"}, Unique: true}
m.Index{Name: "idx_orders_user_date", Fields: []string{"user_id", "created_at"}}
```

## All Operations

### `CreateTable`

Creates a new table with fields and optional indexes.

```go
&m.CreateTable{
    Name: "users",
    Fields: []m.Field{
        {Name: "id",         Type: "integer",   PrimaryKey: true},
        {Name: "email",      Type: "varchar",   Length: 255},
        {Name: "name",       Type: "varchar",   Length: 100, Nullable: true},
        {Name: "created_at", Type: "timestamp", AutoCreate: true},
    },
    Indexes: []m.Index{
        {Name: "idx_users_email", Fields: []string{"email"}, Unique: true},
    },
}
```

**Generated SQL (PostgreSQL):**
```sql
CREATE TABLE users (
    id         INTEGER PRIMARY KEY,
    email      VARCHAR(255) NOT NULL,
    name       VARCHAR(100),
    created_at TIMESTAMP NOT NULL DEFAULT NOW()
);
CREATE UNIQUE INDEX idx_users_email ON users (email);
```

**Down:** `DROP TABLE users`

| Field | Type | Description |
|-------|------|-------------|
| `Name` | `string` | Table name. Required. |
| `Fields` | `[]Field` | Column definitions. |
| `Indexes` | `[]Index` | Indexes to create alongside the table. |

---

### `DropTable`

Drops an existing table.

```go
&m.DropTable{Name: "legacy_sessions"}
```

**Generated SQL:** `DROP TABLE legacy_sessions`

**Down:** Reconstructs `CREATE TABLE` from the pre-drop schema state. Because `Down` needs the original table definition, the schema state at the point of rollback must include it — which it will, as long as you roll back in order.

> **Destructive.** This operation causes data loss. `IsDestructive()` returns `true`.

| Field | Type | Description |
|-------|------|-------------|
| `Name` | `string` | Table name. Required. |

---

### `RenameTable`

Renames a table.

```go
&m.RenameTable{OldName: "accounts", NewName: "users"}
```

**Generated SQL (PostgreSQL):** `ALTER TABLE accounts RENAME TO users`

**Down:** `ALTER TABLE users RENAME TO accounts`

| Field | Type | Description |
|-------|------|-------------|
| `OldName` | `string` | Current table name. |
| `NewName` | `string` | New table name. |

---

### `AddField`

Adds a column to an existing table.

```go
&m.AddField{
    Table: "users",
    Field: m.Field{Name: "phone", Type: "varchar", Length: 20, Nullable: true},
}
```

**Generated SQL:** `ALTER TABLE users ADD COLUMN phone VARCHAR(20)`

**Down:** `ALTER TABLE users DROP COLUMN phone`

| Field | Type | Description |
|-------|------|-------------|
| `Table` | `string` | Table to modify. |
| `Field` | `Field` | Column definition to add. |

> **Tip:** New columns added to existing tables should usually be `Nullable: true` or have a `Default` value, otherwise the `ALTER TABLE` will fail if existing rows exist.

---

### `DropField`

Removes a column from an existing table.

```go
&m.DropField{Table: "users", Field: "legacy_token"}
```

**Generated SQL:** `ALTER TABLE users DROP COLUMN legacy_token`

**Down:** Reconstructs `ADD COLUMN` from the pre-drop schema state.

> **Destructive.** This operation causes data loss. `IsDestructive()` returns `true`.

| Field | Type | Description |
|-------|------|-------------|
| `Table` | `string` | Table to modify. |
| `Field` | `string` | Column name to drop. |

---

### `AlterField`

Changes the definition of an existing column.

```go
&m.AlterField{
    Table: "users",
    OldField: m.Field{Name: "bio", Type: "varchar", Length: 255, Nullable: true},
    NewField: m.Field{Name: "bio", Type: "text",    Nullable: true},
}
```

**Generated SQL (PostgreSQL):** `ALTER TABLE users ALTER COLUMN bio TYPE TEXT`

**Down:** Applies `OldField` definition back to `NewField` column.

Both `OldField` and `NewField` must have the same `Name`. The auto-generator fills these in from the schema diff — when writing by hand, copy the full field definition for both.

| Field | Type | Description |
|-------|------|-------------|
| `Table` | `string` | Table to modify. |
| `OldField` | `Field` | Current column definition (before change). |
| `NewField` | `Field` | Desired column definition (after change). |

---

### `RenameField`

Renames a column.

```go
&m.RenameField{Table: "users", OldName: "fullname", NewName: "display_name"}
```

**Generated SQL (PostgreSQL):** `ALTER TABLE users RENAME COLUMN fullname TO display_name`

**Down:** `ALTER TABLE users RENAME COLUMN display_name TO fullname`

| Field | Type | Description |
|-------|------|-------------|
| `Table` | `string` | Table to modify. |
| `OldName` | `string` | Current column name. |
| `NewName` | `string` | New column name. |

---

### `AddIndex`

Adds an index to an existing table.

```go
&m.AddIndex{
    Table: "orders",
    Index: m.Index{
        Name:   "idx_orders_user_created",
        Fields: []string{"user_id", "created_at"},
        Unique: false,
    },
}
```

**Generated SQL:** `CREATE INDEX idx_orders_user_created ON orders (user_id, created_at)`

**Down:** `DROP INDEX idx_orders_user_created`

| Field | Type | Description |
|-------|------|-------------|
| `Table` | `string` | Table to index. |
| `Index` | `Index` | Index definition. |

---

### `DropIndex`

Removes an index.

```go
&m.DropIndex{Table: "orders", Index: "idx_orders_user_created"}
```

**Generated SQL:** `DROP INDEX idx_orders_user_created`

**Down:** Reconstructs `CREATE INDEX` from the pre-drop schema state.

| Field | Type | Description |
|-------|------|-------------|
| `Table` | `string` | Table the index belongs to. |
| `Index` | `string` | Index name to drop. |

---

### `RunSQL`

Executes raw SQL directly. This is the escape hatch for anything the typed operations cannot express.

```go
&m.RunSQL{
    ForwardSQL:  "UPDATE users SET status = 'active' WHERE status IS NULL",
    BackwardSQL: "UPDATE users SET status = NULL WHERE status = 'active'",
}
```

**Up:** Executes `ForwardSQL` verbatim.

**Down:** Executes `BackwardSQL` verbatim. If rollback is not possible or not needed, leave `BackwardSQL` empty.

**State:** `RunSQL` does **not** update the in-memory schema state. If your SQL creates or alters objects that later operations depend on, use typed operations for those.

| Field | Type | Description |
|-------|------|-------------|
| `ForwardSQL` | `string` | SQL to execute on `up`. |
| `BackwardSQL` | `string` | SQL to execute on `down`. Empty string means no-op on rollback. |

> The field names are `ForwardSQL` / `BackwardSQL` (not `Up` / `Down`) to avoid collision with the `Operation` interface method names.

## Writing Custom Migrations

### Modifying a generated migration

After `makemigrations makemigrations` creates a migration, you can edit it freely before it is applied. Common edits:

**Add a data backfill after a column:**

```go
Operations: []m.Operation{
    &m.AddField{
        Table: "users",
        Field: m.Field{Name: "display_name", Type: "varchar", Length: 100, Nullable: true},
    },
    // Backfill display_name from existing name column
    &m.RunSQL{
        ForwardSQL:  "UPDATE users SET display_name = name WHERE display_name IS NULL",
        BackwardSQL: "",
    },
    // Now make it NOT NULL
    &m.AlterField{
        Table:    "users",
        OldField: m.Field{Name: "display_name", Type: "varchar", Length: 100, Nullable: true},
        NewField: m.Field{Name: "display_name", Type: "varchar", Length: 100},
    },
},
```

**Add a NOT NULL column to an existing table (safe pattern):**

```go
Operations: []m.Operation{
    // Step 1: add nullable
    &m.AddField{
        Table: "orders",
        Field: m.Field{Name: "region", Type: "varchar", Length: 50, Nullable: true},
    },
    // Step 2: populate existing rows
    &m.RunSQL{
        ForwardSQL:  "UPDATE orders SET region = 'unknown' WHERE region IS NULL",
        BackwardSQL: "",
    },
    // Step 3: tighten to NOT NULL
    &m.AlterField{
        Table:    "orders",
        OldField: m.Field{Name: "region", Type: "varchar", Length: 50, Nullable: true},
        NewField: m.Field{Name: "region", Type: "varchar", Length: 50},
    },
},
```

### Writing a migration entirely by hand

Create the file manually in the `migrations/` directory following the naming convention:

```
migrations/0005_seed_permissions.go
```

```go
package main

import m "github.com/ocomsoft/makemigrations/migrate"

func init() {
    m.Register(&m.Migration{
        Name:         "0005_seed_permissions",
        Dependencies: []string{"0004_add_permissions_table"},
        Operations: []m.Operation{
            &m.RunSQL{
                ForwardSQL: `
INSERT INTO permissions (name, description) VALUES
    ('users.read',   'Read user records'),
    ('users.write',  'Create and update user records'),
    ('users.delete', 'Delete user records')
`,
                BackwardSQL: `
DELETE FROM permissions
WHERE name IN ('users.read', 'users.write', 'users.delete')
`,
            },
        },
    })
}
```

### Data migration pattern

Use `RunSQL` for data migrations. Keep schema operations (DDL) in separate migrations from data operations (DML) where possible — this makes rollback cleaner.

```go
// 0006_migrate_user_roles.go
package main

import m "github.com/ocomsoft/makemigrations/migrate"

func init() {
    m.Register(&m.Migration{
        Name:         "0006_migrate_user_roles",
        Dependencies: []string{"0005_add_roles_table"},
        Operations: []m.Operation{
            &m.RunSQL{
                ForwardSQL: `
INSERT INTO user_roles (user_id, role_id)
SELECT u.id, r.id
FROM users u
JOIN roles r ON r.name = u.legacy_role
WHERE u.legacy_role IS NOT NULL
`,
                BackwardSQL: `
DELETE FROM user_roles
WHERE (user_id, role_id) IN (
    SELECT u.id, r.id
    FROM users u
    JOIN roles r ON r.name = u.legacy_role
    WHERE u.legacy_role IS NOT NULL
)
`,
            },
        },
    })
}
```

### Multiple operations per migration

A migration can contain any number of operations. They run in order on `up` and in reverse on `down`:

```go
Operations: []m.Operation{
    &m.CreateTable{
        Name: "posts",
        Fields: []m.Field{
            {Name: "id",         Type: "integer",   PrimaryKey: true},
            {Name: "author_id",  Type: "foreign_key",
                ForeignKey: &m.ForeignKey{Table: "users", OnDelete: "CASCADE"}},
            {Name: "title",      Type: "varchar",   Length: 500},
            {Name: "body",       Type: "text",      Nullable: true},
            {Name: "published",  Type: "boolean",   Default: "false"},
            {Name: "created_at", Type: "timestamp", AutoCreate: true},
            {Name: "updated_at", Type: "timestamp", AutoUpdate: true},
        },
    },
    &m.AddIndex{
        Table: "posts",
        Index: m.Index{Name: "idx_posts_author", Fields: []string{"author_id"}},
    },
    &m.AddIndex{
        Table: "posts",
        Index: m.Index{Name: "idx_posts_published", Fields: []string{"published", "created_at"}},
    },
},
```

## Merge Migrations

When two developers create migrations independently from the same parent, the DAG develops branches:

```
0001_initial
├── 0002_add_phone    (developer A)
└── 0003_add_avatar   (developer B)
```

`makemigrations makemigrations --merge` generates a merge migration automatically:

```go
// 0004_merge_0002_add_phone_and_0003_add_avatar.go
package main

import m "github.com/ocomsoft/makemigrations/migrate"

func init() {
    m.Register(&m.Migration{
        Name:         "0004_merge_0002_add_phone_and_0003_add_avatar",
        Dependencies: []string{"0002_add_phone", "0003_add_avatar"},
        Operations:   []m.Operation{},
    })
}
```

A merge migration:
- Has **empty operations** — it does nothing to the database
- Has **two or more dependencies** — pointing at each branch leaf
- Exists solely to create a common ancestor in the DAG

After a merge migration, subsequent migrations have a single node to depend on again.

## Squash Migrations

Squash migrations replace a sequence of old migrations with a single equivalent migration. This keeps the `migrations/` directory manageable as a project grows.

`makemigrations makemigrations --squash` (not yet a CLI flag — generate manually or use the `SquashGenerator` API) produces a file like:

```go
package main

import m "github.com/ocomsoft/makemigrations/migrate"

func init() {
    m.Register(&m.Migration{
        Name:         "0001_squashed_0005",
        Dependencies: []string{},
        Replaces:     []string{"0001_initial", "0002_add_phone", "0003_add_avatar", "0004_merge", "0005_add_roles"},
        Operations: []m.Operation{
            // All operations from 0001–0005, consolidated
            &m.CreateTable{ /* ... */ },
            &m.AddField{    /* ... */ },
            // ...
        },
    })
}
```

**`Replaces`** tells the runner that if all the named migrations are already applied in the history table, the squash migration itself should be considered applied too (no re-run needed). This allows gradual cut-over without downtime.

> For new databases, only the squash migration runs. For existing databases with the original migrations already applied, the squash is automatically treated as applied.

## Previewing SQL Before Applying

Before running `up`, preview exactly what SQL will execute:

```bash
cd migrations && go build -o migrate .
./migrate showsql
```

Output:

```sql
-- 0005_seed_permissions
INSERT INTO permissions (name, description) VALUES
    ('users.read',   'Read user records'),
    ...

-- 0006_migrate_user_roles
INSERT INTO user_roles (user_id, role_id)
SELECT u.id, r.id FROM users u ...
```

This is useful for auditing, security review, and debugging provider-specific SQL generation. No database connection is needed — SQL is generated in-process.

## File Naming Convention

Generated files follow the pattern:

```
NNNN_description.go
```

- `NNNN` — zero-padded 4-digit sequence number (0001, 0002, ..., 9999)
- `description` — lowercase, words joined by underscores

The sequence number determines registration order when two migrations have the same parents. It does **not** imply a strict linear order — the DAG is determined solely by `Dependencies`, not file names.

Examples:
```
0001_initial.go
0002_add_phone.go
0003_add_avatar.go
0004_merge_0002_add_phone_and_0003_add_avatar.go
0005_add_roles.go
0006_seed_permissions.go
```

## Registration and the init() Pattern

The `Register()` function adds a migration to the global registry. It panics (intentionally) if:

- A `nil` migration is passed
- A migration with an empty `Name` is passed
- A duplicate `Name` is registered

These panics happen at binary startup, before any command runs, making misconfigured migration sets immediately visible.

```go
// Global registry — populated by all init() functions
func Register(m *Migration)

// Used for testing with an isolated registry
func NewRegistry() *Registry
func (r *Registry) Register(m *Migration)
```

Because Go guarantees `init()` functions run before `main()`, every migration is registered before the CLI processes any command. The order of `init()` calls follows Go's package initialisation rules (alphabetical file order within a package).

## Common Patterns Reference

### Timestamps

```go
{Name: "created_at", Type: "timestamp", AutoCreate: true},
{Name: "updated_at", Type: "timestamp", AutoUpdate: true},
```

### Soft delete

```go
{Name: "deleted_at", Type: "timestamp", Nullable: true},
```

### UUID primary key

```go
{Name: "id", Type: "uuid", PrimaryKey: true, Default: "new_uuid"},
```

### Foreign key with cascade

```go
{
    Name: "user_id",
    Type: "foreign_key",
    ForeignKey: &m.ForeignKey{Table: "users", OnDelete: "CASCADE"},
},
```

### Decimal money field

```go
{Name: "amount", Type: "decimal", Precision: 10, Scale: 2},
```

### Unique composite index

```go
m.Index{
    Name:   "idx_memberships_user_org",
    Fields: []string{"user_id", "org_id"},
    Unique: true,
},
```

### Conditional RunSQL (no-op down)

```go
&m.RunSQL{
    ForwardSQL:  "UPDATE config SET value = 'v2' WHERE key = 'api_version'",
    BackwardSQL: "", // intentionally irreversible
},
```
